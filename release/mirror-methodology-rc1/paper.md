# MIRROR: Local Evidence Bundles for Reproducible Review, Not Certification

Private release candidate from Tyche Institute, Tallinn, Estonia. Not peer reviewed. Not externally published. Not a legal compliance opinion.

Status: local release-candidate folder for human review before any public action.

## Abstract

MIRROR is a lightweight method for keeping research artifacts reviewable as they move through iterative drafting, automated workers, and later human evaluation. A MIRROR bundle places an artifact, source copies or source descriptors, structured claim records, notes, assumptions, and SHA-256 digests into a local directory that can be replayed offline. The aim is narrow but useful: reduce the cost of asking which bytes were reviewed, which sources were retained, which claims were made, and which limitations were known at the time of bundling. MIRROR does not decide whether a paper is true, complete, novel, legally compliant, peer reviewed, authored by a particular person, externally timestamped, or suitable for public release. This release candidate presents the v0.1 workflow as a conservative provenance substrate for scholarly maintenance and AI-governance research.

## 1. Problem

Research artifacts often drift. A draft is edited, a table is regenerated, a source page changes, or a worker leaves behind a note that is hard to connect to the final text. Later review then starts with avoidable uncertainty: what exact object was inspected, where did its evidence come from, and which claims depended on which files?

This uncertainty is especially costly in AI-governance and evidence-engineering work, where cautious wording matters. A small change from "this file was retained and can be rehashed" to "this claim is proven" changes the nature of the statement. MIRROR is designed to keep that boundary visible.

## 2. Bundle format

A MIRROR v0.1 bundle is a directory adjacent to or inside a research artifact package. It normally contains:

- `manifest.json`, describing the artifact, sources, assumptions, creation metadata, and non-signing boundary;
- `claims.json`, listing claim text, evidence pointers, risk levels, and safe wording;
- `notes.md`, explaining how to read the bundle and what it does not establish;
- `artifact/`, containing the bundled artifact copy when generated by the current reference bundler;
- `sources/`, containing local source copies or URL descriptor records.

The central design choice is separation. The artifact is not treated as self-explanatory. Claims are not hidden in prose only. Source records are not assumed to establish truth. Each relationship is made explicit enough that a reviewer can challenge it.

## 3. Verifier semantics

The reference verifier asks a mechanical question: do the retained local files match the declared schema and SHA-256 digests? A green result supports local byte and schema replay. A failing result identifies maintenance work such as a missing file, malformed JSON, unsafe pointer, duplicate identifier, or digest mismatch.

Neither result is a scholarly verdict. A green result can coexist with weak evidence, stale sources, incomplete claims, or an unpublishable draft. A failure can reflect ordinary editing that has not yet been re-bundled. MIRROR therefore reports local consistency rather than a grade.

## 4. Worked example

This release candidate includes `examples/minimal-artifact.md` and `examples/minimal-source.txt`. The example demonstrates one low-risk claim: a retained source file contains a short statement about a local replay boundary. The example bundle can be verified offline with the same reference verifier used for the paper bundle.

The example is deliberately small. Its purpose is to show the relationship between artifact, source, claim, and note. It is not evidence that MIRROR is complete, secure against all attacks, or suitable as a regulated assurance system.

## 5. Daily roll-up and local anchor chain

The broader MIRROR workspace also experiments with daily roll-ups and local anchor records. Those records can help detect local drift across a changing research workspace by preserving digests, Merkle roots, and previous-record links. In this release candidate, they are presented as continuity aids only. They are not external timestamps, notarizations, publication decisions, audits, or proof that sibling research lanes are complete.

## 6. Threat model

MIRROR is meant to catch ordinary provenance failures: missing source files, stale digests, ambiguous claim pointers, schema drift, and unsafe assurance wording. It assumes that the local filesystem and retained files are available for review. It does not solve identity proof, key custody, source truth, external time, malicious operating-system behavior, licensing clearance, peer review, or legal interpretation.

## 7. Limitations

The main limitation is scope. SHA-256 equality is powerful for identifying bytes and weak for interpreting meaning. A verifier can show that a retained source copy has not changed inside the bundle; it cannot show that the source was authoritative, current, complete, or fairly interpreted. A claim register can make review easier; it cannot force the author to list every relevant claim. A local anchor can make continuity easier to inspect; it cannot provide independent public time.

## 8. Review protocol

A reviewer should start with the mechanical check, then move to claim challenge:

1. Verify the bundle offline.
2. Choose one claim from `claims.json`.
3. Open the evidence pointer and confirm what it actually supports.
4. Compare the claim text with the safe wording.
5. Record whether the claim should be kept, narrowed, split, or blocked.

The useful output of MIRROR is not a badge. It is a shorter path from artifact to evidence to challenge.

## 9. Public-release boundary

This folder is a private release candidate. It may be copied, rewritten, or reviewed locally. It has not been pushed, deposited, signed, timestamped, peer reviewed, or published. Any public release still requires human review for private paths, licensing, citation quality, overclaiming, employer-boundary issues, and strategic fit.
